## Learnings

### Architecture Review (2025)

**Solution structure:** 6 projects in `src/Refitter.slnx`. Clean separation: `Refitter.Core` (netstandard2.0) is the engine, `Refitter` (CLI, net8.0/9.0/10.0) is one consumer, `Refitter.SourceGenerator` (netstandard2.0, IIncrementalGenerator) is another, and `Refitter.MSBuild` (netstandard2.0) shells out to the CLI via `Process.Start`. Tests split into `Refitter.Tests` (core) and `Refitter.SourceGenerator.Tests`.

**Code generation pipeline (end-to-end):**
1. `OpenApiDocumentFactory.CreateAsync` loads/merges OpenAPI specs (supports files, URLs, YAML/JSON, multi-file merge, external `$ref` via OasReader fallback to NSwag).
2. `RefitGenerator.CreateAsync` runs tag/path/schema filters on the loaded `OpenApiDocument`.
3. `RefitGenerator.Generate()` (or `GenerateMultipleFiles()`) creates a `CSharpClientGeneratorFactory` â†’ `CustomCSharpClientGenerator` (NSwag wrapper), then selects one of three `IRefitInterfaceGenerator` implementations based on `MultipleInterfaces` setting.
4. Interface generators (`RefitInterfaceGenerator`, `RefitMultipleInterfaceGenerator`, `RefitMultipleInterfaceByTagGenerator`) iterate paths/operations, use `ParameterExtractor` for params, `XmlDocumentationGenerator` for docs, and emit `GeneratedCode` records.
5. Contract generation delegated entirely to NSwag `generator.GenerateFile()`.
6. DI registration code produced by `DependencyInjectionGenerator` or `ApizrRegistrationGenerator`.

**Key abstractions:**
- `IRefitInterfaceGenerator` â€” strategy interface for single vs multi-interface generation. Three implementations share a base class (`RefitInterfaceGenerator`).
- `RefitGeneratorSettings` â€” the central configuration object (~50 properties), consumed by Core, CLI, SourceGenerator, and MSBuild.
- `CSharpClientGeneratorFactory` â€” factory bridging Refitter settings to NSwag's `CSharpClientGeneratorSettings`. Uses reflection-based property mapping (`MapCSharpGeneratorSettings`).
- `ParameterExtractor` â€” static class extracting route/query/body/header/form/binary params from NSwag's `CSharpOperationModel`.
- `OperationNameGenerator` â€” adapter wrapping multiple NSwag `IOperationNameGenerator` implementations with kebab/snake/space/colon sanitization.

**SourceGenerator:** Uses `IIncrementalGenerator` pattern, reads `.refitter` additional text files, deserializes to `RefitGeneratorSettings`, calls `RefitGenerator` synchronously (`.GetAwaiter().GetResult()`), writes output to disk. Shares Core source files via `<Compile Include>` (not a project reference).

**MSBuild task:** Does NOT reference Refitter.Core. Instead shells out to `dotnet refitter.dll` as a child process, parses `.refitter` JSON with regex to predict output files.

**Settings design:** CLI `Settings.cs` (Spectre.Console `CommandSettings`) maps 1:1 to `RefitGeneratorSettings` via `CreateRefitGeneratorSettings()` in `GenerateCommand.cs`. Many boolean CLI flags are inverted (e.g., `--no-auto-generated-header` â†’ `AddAutoGeneratedHeader = !settings.NoAutoGeneratedHeader`).

**Notable strengths:**
- Clean Core/consumer separation enables three delivery mechanisms from one engine.
- Strategy pattern for interface generation modes is well-designed and extensible.
- Comprehensive XML doc generation with status code documentation.
- Schema tree-shaking (`SchemaCleaner`) is thorough with inheritance hierarchy support.
- `partial interface` generation allows user extension.

**Architectural concerns identified:**
1. **SourceGenerator sync-over-async:** `RefitGenerator.CreateAsync(...).GetAwaiter().GetResult()` in source generator can deadlock in certain IDE hosts.
2. **Duplicated source compilation:** SourceGenerator includes Core `.cs` files via `<Compile Include>` rather than IL reference â€” changes to Core files compile twice and can drift.
3. **MSBuild JSON parsing with regex:** `RefitterGenerateTask` uses regex to parse JSON config (`ExtractJsonStringValue`/`ExtractJsonBoolValue`) instead of a JSON parser, which is fragile.
4. **God settings class:** `RefitGeneratorSettings` has ~50 properties and growing, with no grouping beyond `NamingSettings`, `DependencyInjectionSettings`, `ApizrSettings`, and `CodeGeneratorSettings`.
5. **Reflection-based settings mapping:** `CSharpClientGeneratorFactory.MapCSharpGeneratorSettings` uses reflection to copy properties between Refitter and NSwag settings â€” fragile across NSwag version upgrades.
6. **No interface for `RefitGenerator`:** The public entry point `RefitGenerator` is a concrete class with no interface, limiting testability and extensibility.
7. **Large `ParameterExtractor`:** 470+ line static class handling all parameter kinds â€” could benefit from decomposition.
8. **`GenerateCommand` is 500+ lines** mixing orchestration, file I/O, console output, analytics, validation, and settings mapping.

### Issue #944 Review â€” Non-ASCII in XML Status-Code Comments (2026-03-06)

**Reviewed and APPROVED** the patch for issue #944. Root cause: NSwag's `CSharpResponseModel.ExceptionDescription` calls `ConversionUtilities.ConvertToStringLiteral()` which encodes non-ASCII characters as `\uXXXX` C# string literal escapes. The fix adds a `DecodeJsonEscapedText()` method in `XmlDocumentationGenerator` that decodes these before XML-escaping. Key validation points:

1. **Scope is correctly narrow:** Only `ExceptionDescription` uses `ConvertToStringLiteral`; other NSwag model fields (Summary, Description) surface raw strings. No other call sites need the fix.
2. **Ordering matters:** `EscapeSymbols(DecodeJsonEscapedText(...))` â€” decode first, then XML-escape â€” prevents decoded `<`, `>`, `&` from breaking XML.
3. **Bounds checking correct:** `index + 4 < input.Length` guard on `\uXXXX` parsing matches `Substring(index + 1, 4)` requirements.
4. **Surrogate pairs work:** Each `\uXXXX` decoded to `(char)` is valid C# UTF-16, so surrogate pairs naturally compose.
5. **Tests prove the fix:** Unit test against generator directly + integration test from YAML spec through full pipeline + compilation check. All 1415 tests pass.

**Key learning:** When NSwag model properties use `ConvertToStringLiteral` (designed for C# code generation), the output contains escape sequences unsuitable for XML documentation. Any future properties that exhibit this pattern will need similar decode-before-escape treatment.
